## Story

As a **platform operator deploying Tamma as a SaaS service**,
I want GitHub App authentication with installation lifecycle management,
so that multiple organizations can install Tamma from the GitHub Marketplace and receive automated development workflows without sharing personal access tokens.

## Overview

Implement dual authentication support for the GitHub platform: the existing PAT-based auth for self-hosted deployments, and new GitHub App auth for SaaS deployments. GitHub App auth uses a two-tier model — a short-lived JWT (signed with the App's RSA private key) authenticates as the App itself, then per-installation tokens (scoped to specific org/repo installations) are obtained via the GitHub API. Installation tokens auto-expire after 1 hour and must be refreshed transparently.

This story also covers the installation callback flow (the redirect endpoint GitHub calls after a user installs the App), the database schema for persisting installations, and secure private key management via environment variables or file paths.

## Acceptance Criteria

1. GitHub platform supports dual auth modes: PAT (existing) and GitHub App (new), selected via configuration
2. GitHub App auth generates JWT signed with RSA private key using `@octokit/auth-app`
3. Installation tokens are auto-refreshed before expiration (with configurable buffer, default 5 minutes)
4. Installation callback endpoint (`GET /api/github/callback`) handles post-install redirect, persists installation to database
5. Installation database schema stores: `installation_id`, `account_login`, `account_type` (org/user), `app_id`, `permissions`, `repository_selection`, `created_at`, `suspended_at`
6. Private key can be provided via environment variable (`GITHUB_APP_PRIVATE_KEY`) or file path (`GITHUB_APP_PRIVATE_KEY_PATH`)
7. GitHub App configuration validated at startup — missing `appId`, `privateKey`, or `webhookSecret` produces clear error messages
8. Platform factory creates correct platform instance based on auth mode in config
9. Unit tests cover: JWT generation, token refresh, installation persistence, callback handling, config validation, factory selection
10. Integration tests cover: end-to-end installation callback flow, token lifecycle with mock GitHub API

### Requirements Context Summary

**Epic Context:** This story is the foundational enabler for Tamma's SaaS deployment model. Without GitHub App auth, every user would need to create and manage their own PAT, which doesn't scale for a marketplace-style offering. GitHub App auth provides: org-level installation (not personal), fine-grained permissions, automatic token management, and webhook delivery tied to the App.

**Technical Context:** GitHub Apps use a two-tier authentication model:
1. **App-level JWT**: Signed with RSA private key, expires in 10 minutes. Used for App-level API calls (list installations, get installation details).
2. **Installation token**: Obtained by exchanging App JWT for an installation-scoped token. Expires in 1 hour. Used for all repo-level API calls (create PRs, read issues, push code).

The `@octokit/auth-app` library handles both tiers transparently, but we must ensure token refresh happens proactively (not on 401 failure).

**Integration Points:**
- Story 1.5-4 (Web Server): Provides the Fastify server for the callback endpoint
- Story 1.5-6 (Webhooks): Webhook handler must route `installation` events to the installation store
- Story 1.5-12 (SaaS Coordinator): Depends on this story for installation discovery
- Story 1-5 (GitHub Platform): Existing PAT-based implementation extended, not replaced

### Implementation Guidance

**Key Design Decisions:**

- **Dual auth, not replacement**: PAT auth remains for self-hosted users. Auth mode selected via config, not runtime detection.
- **`@octokit/auth-app` over manual JWT**: The official library handles JWT signing, installation token caching, and refresh. No reason to reimplement.
- **Private key as env var**: For container/SaaS deployments where file mounts are inconvenient. The env var holds the PEM-encoded key directly.
- **Installation DB over in-memory**: Installations must survive restarts. The SaaS coordinator (Story 1.5-12) queries this table to discover which repos to serve.

**Technical Specifications:**

- **Auth library**: `@octokit/auth-app` (official GitHub auth strategy)
- **JWT lifetime**: 10 minutes (GitHub maximum)
- **Installation token lifetime**: 1 hour (GitHub fixed)
- **Token refresh buffer**: 5 minutes before expiry (configurable)
- **Database**: PostgreSQL `github_installations` table (see Task 4)

**Testing Strategy:**

- Unit tests mock `@octokit/auth-app` to verify our wrapper logic (refresh timing, error handling)
- Integration tests use MSW to mock GitHub API responses for the callback flow
- No real GitHub API calls in CI — use test fixtures from GitHub's webhook payload examples

### Risks and Mitigations

| Risk | Severity | Mitigation |
| ---- | -------- | ---------- |
| Private key exposure in environment variable | High | Document secure env var injection (Docker secrets, K8s secrets). Never log the key. |
| Installation token refresh race condition under high concurrency | Medium | Use `@octokit/auth-app`'s built-in caching which handles concurrent refresh requests |
| GitHub App rate limits differ from PAT rate limits (5000/hr per installation vs 5000/hr per user) | Low | Document rate limit differences. Installation tokens have per-installation limits which is actually better for multi-tenant. |
| Migration conflicts with existing database schema | Medium | Use sequential migration numbering. Test migration up/down in CI. |
