## Story

As a **SaaS platform operator**,
I want webhook events and task queue entries tagged and routed by installation ID,
so that each GitHub App installation's work is isolated, correctly routed to the right engine, and auditable per-tenant.

## Overview

In Tamma's SaaS model, a single orchestrator receives webhook events from many GitHub App installations. Each webhook payload includes an `installation.id` field identifying which installation generated the event. This story ensures that:

1. **Webhook events are tagged** with `installation_id` at the point of receipt
2. **Task queue entries** include `installation_id` as a partition key for routing
3. **Webhook routing** delivers events to the correct engine (matched by installation ID via the SaaS Coordinator's engine registry)
4. **Per-tenant isolation** prevents one installation's tasks from being processed by another installation's engine

This is the glue between Stories 1.5-6 (webhooks), 1.5-12 (coordinator), and the task queue system.

## Acceptance Criteria

1. All webhook events from GitHub App installations include `installation_id` in the task payload
2. Task queue entries are partitioned by `installation_id` — tasks are only dequeued by the matching installation's engine
3. Webhook routing resolves `installation_id` → engine via the coordinator's engine registry, returning 404 for unknown installations
4. Installation lookup (from webhook `installation.id` to DB installation record) completes in < 10ms (cached)
5. Per-tenant task isolation: a task enqueued for installation A is never dequeued by installation B
6. Task queue schema includes `installation_id` column with index for partition queries
7. Unit tests cover: webhook tagging, task routing, installation lookup, isolation guarantees
8. Integration tests cover: end-to-end webhook → task queue → engine routing for multiple concurrent installations

### Requirements Context Summary

**Epic Context:** Multi-tenant task routing is the last piece of the SaaS puzzle. Without it, webhooks from different installations would all land in the same undifferentiated task queue, and any engine could pick up any task — breaking tenant isolation. This story ensures that the task queue is partitioned by installation, giving each tenant an isolated work queue.

**Technical Context:** GitHub webhook payloads for GitHub App installations always include an `installation` object with an `id` field. This is the natural partition key for multi-tenant routing. The task queue already exists (from Story 1.5-3/1.5-6), but it's currently single-tenant — tasks are dequeued FIFO with no tenant awareness.

**Integration Points:**
- Story 1.5-6 (Webhook Integration): Webhook handler extracts `installation_id` and includes it in task payloads
- Story 1.5-12 (SaaS Coordinator): Engine registry maps `installation_id` to active engines
- Story 1.5-11 (GitHub App Integration): `IInstallationStore` provides installation lookup
- Story 1.5-3 (Service Mode): Task queue is extended with `installation_id` partitioning

### Implementation Guidance

**Key Design Decisions:**

- **Nullable `installation_id`**: The column is nullable to maintain backward compatibility with self-hosted mode. In self-hosted mode, all tasks have `installation_id = NULL` and the single engine dequeues all tasks. In SaaS mode, tasks without `installation_id` are rejected.
- **Cache for installation lookup**: Webhook handlers may receive bursts of events for the same installation. Caching the `installationId → installation` lookup with a 60-second TTL avoids repeated DB queries while staying reasonably fresh.
- **Partition at query level, not separate tables**: Using `WHERE installation_id = $1` in the dequeue query is simpler than creating per-tenant tables or schemas. PostgreSQL handles this efficiently with a btree index on `installation_id`.
- **Defense-in-depth isolation**: Even though the dequeue query filters by `installation_id`, we validate the dequeued task's `installation_id` matches at the application level. This catches bugs in query construction.

**Technical Specifications:**

- **Task queue column**: `installation_id BIGINT NULL REFERENCES github_installations(installation_id)`
- **Index**: `CREATE INDEX idx_tasks_installation ON task_queue(installation_id) WHERE status = 'pending'`
- **Cache TTL**: 60 seconds (configurable)
- **Lookup latency target**: < 10ms (with cache hit)
- **Backward compatibility**: Self-hosted mode ignores `installation_id` entirely

**Testing Strategy:**

- Unit tests use in-memory task queue and mock installation store
- Integration tests create 3 installations, enqueue tasks for each, verify strict isolation
- Performance tests verify cached lookup latency < 10ms
- Backward compatibility tests verify self-hosted mode works with null `installation_id`

### Risks and Mitigations

| Risk | Severity | Mitigation |
| ---- | -------- | ---------- |
| Webhook storm from a single installation could starve other tenants | High | Per-installation rate limiting at the webhook handler level. Fair queuing (round-robin dequeue across installations). |
| Cache staleness causes routing to removed installations | Low | 60-second TTL is short enough. Explicit cache invalidation on `installation.deleted` webhook event. |
| Migration adds non-null column to existing task queue with data | Medium | Column is nullable. No data migration needed — existing rows get NULL (self-hosted mode). |
| Cross-tenant task leakage due to query bugs | High | Defense-in-depth: application-level validation after dequeue. Isolation violation logged as ERROR with alerting. |
