## Story

As a **user who has installed the Tamma GitHub App**,
I want Tamma to automatically provision an API key into my repository's GitHub Actions secrets during setup,
so that my Actions-based workers can authenticate to the Tamma SaaS API for LLM access, workflow status updates, and result reporting — without me configuring any API keys manually.

## Overview

When a user installs the Tamma GitHub App, the installation callback (Story 1.5-11) already stores the installation and its repos in the database. This story extends that flow to also:

1. **Generate a per-installation API key** — a cryptographically random token that identifies and authenticates this installation's workers
2. **Store the key hash** in the database (never store plaintext API keys)
3. **Write `TAMMA_API_KEY` to each repo's GitHub Actions secrets** — using the GitHub REST API with libsodium sealed-box encryption
4. **Expose SaaS API endpoints** — that workers call using this key for LLM inference, workflow status, and result callbacks
5. **Handle key lifecycle** — rotation on demand, revocation on installation removal, re-provisioning when repos are added

The result: users install the Tamma GitHub App and their repos are immediately ready to run Tamma workers in GitHub Actions. Zero manual configuration.

## Acceptance Criteria

1. Installation callback generates a per-installation API key (256-bit, URL-safe base64) and stores its SHA-256 hash in the `github_installations` table
2. API key is written as `TAMMA_API_KEY` secret to every repo in the installation using GitHub's encrypted secrets API (`PUT /repos/{owner}/{repo}/actions/secrets/{secret_name}`)
3. Secret encryption uses `libsodium-wrappers` sealed-box (`crypto_box_seal`) with the repo's public key retrieved from `GET /repos/{owner}/{repo}/actions/secrets/public-key`
4. When `installation_repositories.added` webhook fires, `TAMMA_API_KEY` is provisioned to the newly added repos
5. When installation is deleted, the API key hash is removed from the database (GitHub deletes the secrets automatically when the App is uninstalled)
6. API key validation endpoint (`POST /api/v1/auth/validate`) returns installation context (installationId, accountLogin, permissions) for a valid key, 401 for invalid
7. Tamma SaaS API endpoints authenticated by `TAMMA_API_KEY` via `Authorization: Bearer <key>` header:
   - `POST /api/v1/llm/chat` — proxied LLM inference (provider selection, model, messages)
   - `POST /api/v1/workflows/{id}/status` — workflow progress updates
   - `POST /api/v1/workflows/{id}/result` — final result callback
8. LLM proxy uses the SaaS operator's provider keys (from server config), not the user's — users never need LLM API keys
9. API key can be rotated via `POST /api/v1/installations/{id}/rotate-key` (generates new key, updates all repo secrets, invalidates old key)
10. GitHub App requires `secrets: write` permission (added to App manifest)
11. Unit tests cover: key generation, hashing, secret encryption, API key validation, LLM proxy routing
12. Integration tests cover: end-to-end callback → key generation → secret provisioning → API call with key

### Requirements Context Summary

**Epic Context:** This story is the linchpin that makes the SaaS model frictionless. Without it, users would need to manually configure LLM API keys in their repos — a dealbreaker for adoption. By generating and provisioning a single `TAMMA_API_KEY`, users get a zero-config experience: install the App, and their repos are immediately ready.

**Technical Context:** GitHub's Actions secrets API requires secrets to be encrypted client-side using the repo's public key and libsodium's sealed-box algorithm. The flow is:
1. `GET /repos/{owner}/{repo}/actions/secrets/public-key` → returns `{ key, key_id }`
2. Encrypt the secret using `sodium.crypto_box_seal(secret, publicKey)`
3. `PUT /repos/{owner}/{repo}/actions/secrets/TAMMA_API_KEY` with `{ encrypted_value, key_id }`

The GitHub App needs `secrets: write` permission to do this. This permission is set in the App's configuration on GitHub.

### Implementation Guidance

**Key Design Decisions:**

- **Hash-only storage**: The plaintext API key is NEVER stored in the database. Only the SHA-256 hash is stored. The plaintext is shown to the user exactly once (at installation time) and written to GitHub secrets. If the user loses it, they must rotate (generate a new one).

- **Per-installation, not per-repo**: One API key per installation (org/user), shared across all repos in that installation. This simplifies management — when a new repo is added, the same key is provisioned. The installation is the tenant boundary.

- **Re-provisioning on repo add**: When `installation_repositories.added` fires, we need the plaintext key to write to the new repo's secrets. Two options:
  - **Option A**: Store the plaintext key encrypted with a server-side master key (AES-256-GCM). This allows re-provisioning without user interaction.
  - **Option B**: Store only the hash. When repos are added, generate a new key (rotate), provision to ALL repos, invalidate old key.
  - **Recommended: Option A** — better UX (no surprise key rotation), and the master key is already needed for other secrets (see CLAUDE.md: "API keys encrypted at rest (AES-256)").

- **`libsodium-wrappers` over `tweetnacl`**: `tweetsodium` (GitHub's wrapper) is deprecated. `libsodium-wrappers` is the official JS binding, actively maintained, faster, and provides `crypto_box_seal()` directly.

- **LLM proxy over direct keys**: The Tamma API proxies LLM calls using the SaaS operator's own provider keys. Users never need LLM API keys. This enables:
  - Usage tracking per installation
  - Budget enforcement per tenant
  - Provider switching without touching user repos
  - Centralized rate limiting and error handling

**Technical Specifications:**

- **API key format**: 256-bit random, base64url encoded (43 chars, e.g., `tamma_sk_a1b2c3d4...`)
- **Key prefix**: `tamma_sk_` (8 chars) for identification
- **Hash algorithm**: SHA-256 (hex, 64 chars)
- **Secret name**: `TAMMA_API_KEY` (in GitHub Actions secrets)
- **Encryption**: libsodium `crypto_box_seal()` with repo's X25519 public key
- **Auth header**: `Authorization: Bearer tamma_sk_a1b2c3d4...`
- **Rate limit**: 1000 req/min per API key (configurable)
- **API key column**: `api_key_hash TEXT` (nullable, for self-hosted installations without API keys)
- **Encrypted key column**: `api_key_encrypted TEXT` (AES-256-GCM encrypted plaintext, for re-provisioning)

**Database Migration (003_api_keys.sql):**
```sql

  ADD COLUMN api_key_hash TEXT,
```

### Risks and Mitigations

| Risk | Severity | Mitigation |
| ---- | -------- | ---------- |
| GitHub App `secrets:write` permission may concern users | High | Document clearly in App description. This permission is used ONLY to write `TAMMA_API_KEY`, not to read existing secrets. GitHub's API does not allow reading secret values. |
| API key leaked in workflow logs | Medium | GitHub automatically masks `${{ secrets.* }}` values in logs. Additionally, the key has a recognizable prefix (`tamma_sk_`) for secret scanning tools to flag. |
| Race condition: repos added while provisioning is in progress | Low | Provisioning is idempotent — re-running for an already-provisioned repo just updates the secret. Use `PUT` (create or update). |
| `libsodium-wrappers` adds ~200KB to bundle | Low | Only used server-side in `@tamma/api`, not in CLI or worker bundles. Acceptable. |
| Master encryption key for stored API keys must be managed | Medium | Use `TAMMA_MASTER_KEY` env var (or `TAMMA_MASTER_KEY_FILE` for Docker secrets). Required for SaaS mode, not needed for self-hosted. |
| LLM proxy becomes a single point of failure | Medium | Standard high-availability patterns: multiple API server replicas behind load balancer. LLM proxy is stateless — any replica can handle any request. |
