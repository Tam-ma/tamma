## Story

As a **SaaS platform operator**,
I want a coordinator that discovers GitHub App installations and dispatches work to GitHub Actions runners in each user's repository,
so that Tamma can serve multiple organizations from a single orchestrator instance without requiring each user to self-host.

## Overview

The SaaS Coordinator is the multi-tenant control plane for Tamma's SaaS deployment model. It runs as a long-lived process within the orchestrator, periodically querying the installation database (populated by Story 1.5-11's callback handler and Story 1.5-6's webhook handler) to discover active installations. For each installation, it can dispatch work to GitHub Actions runners in the user's repository via `workflow_dispatch` events, using the installation's scoped token.

The coordinator follows a reconciliation loop pattern (similar to Kubernetes controllers): it compares desired state (all active, non-suspended installations should have engines available) with actual state (which installations currently have running engines), and takes corrective action (start new engines, prune removed ones).

## Acceptance Criteria

1. Coordinator discovers active installations from the `github_installations` database table on startup
2. Coordinator dispatches `workflow_dispatch` events to user repositories to trigger GitHub Actions-based Tamma workers
3. Coordinator handles installation lifecycle events: new installations start engines, removed installations prune engines, suspended installations pause engines
4. Reconciliation loop runs on configurable interval (default 60 seconds), comparing DB state with in-memory engine registry
5. Coordinator uses installation-scoped tokens (from Story 1.5-11) for all per-installation API calls
6. Coordinator emits DCB events for all lifecycle actions: `INSTALLATION.ENGINE_STARTED`, `INSTALLATION.ENGINE_STOPPED`, `INSTALLATION.SUSPENDED`, `INSTALLATION.REMOVED`
7. Graceful shutdown: coordinator stops reconciliation, waits for in-flight dispatches (max 30s timeout), then exits
8. Unit tests cover: discovery, dispatch, reconciliation, lifecycle handling, event emission
9. Integration tests cover: end-to-end coordinator startup → discovery → dispatch flow with mock installation store

### Requirements Context Summary

**Epic Context:** The SaaS Coordinator is the central nervous system of Tamma's multi-tenant SaaS deployment. It bridges the gap between "we have installations in a database" (Story 1.5-11) and "we need to run Tamma engines for each installation" (Story 1.5-13). Without the coordinator, each installation would need manual engine provisioning.

**Technical Context:** The coordinator follows the **reconciliation loop** pattern from Kubernetes controllers:
1. Observe: Query installation store for current desired state
2. Diff: Compare desired state with actual state (in-memory engine registry)
3. Act: Start/stop/suspend engines to match desired state
4. Emit: Record all state transitions as DCB events for audit trail

This pattern is inherently idempotent — running the reconciliation loop multiple times with the same state produces no side effects.

**Integration Points:**
- Story 1.5-11 (GitHub App Integration): Provides `IInstallationStore` for installation discovery
- Story 1.5-6 (Webhooks): Webhook handler updates installation store on lifecycle events, coordinator picks up changes on next reconciliation tick
- Story 1.5-13 (GitHub Actions Worker): The coordinator dispatches work via `workflow_dispatch` to Actions workers
- Story 1-8 (Hybrid Architecture): Coordinator extends the orchestrator's responsibilities in SaaS mode

### Implementation Guidance

**Key Design Decisions:**

- **Reconciliation over event-driven**: The coordinator polls the installation store instead of reacting to webhook events directly. This is simpler, idempotent, and self-healing (if a webhook is missed, the next reconciliation tick catches it). The polling interval (60s) is fast enough for installation lifecycle events.
- **In-memory engine registry**: The coordinator maintains an in-memory map of `installationId → EngineState`. This is rebuilt from the DB on startup, so it's safe to restart the coordinator at any time.
- **Dispatch via workflow_dispatch**: Rather than running engines directly, the coordinator triggers GitHub Actions workflows in user repos. This leverages GitHub's compute, avoids multi-tenant container orchestration complexity, and gives each installation its own isolated runner.
- **Event emission for audit**: All coordinator actions are recorded as DCB events, enabling time-travel debugging of multi-tenant issues.

**Technical Specifications:**

- **Reconciliation interval**: Configurable, default 60 seconds
- **Shutdown timeout**: 30 seconds maximum wait for in-flight operations
- **Engine states**: `available`, `dispatching`, `suspended`, `error`
- **Dispatch inputs**: `installation_id`, `issue_number`, `callback_url`, `orchestrator_version`

**Testing Strategy:**

- Unit tests use `InMemoryInstallationStore` and mock Octokit
- Integration tests verify full lifecycle with simulated installation changes
- Concurrency tests verify no duplicate dispatches during parallel reconciliation

### Risks and Mitigations

| Risk | Severity | Mitigation |
| ---- | -------- | ---------- |
| Reconciliation loop becomes slow with many installations (1000+) | Medium | Batch DB queries, parallelize dispatch operations, add pagination to `listActive()` |
| `workflow_dispatch` rate limits (GitHub allows ~700/hour per installation) | Medium | Implement dispatch queue with rate limiting per installation. Track dispatch counts. |
| Coordinator restart causes duplicate dispatches for in-flight work | Low | `workflow_dispatch` is idempotent from GitHub's perspective — duplicate triggers create separate workflow runs, but the worker checks for in-progress work before starting |
| Installation store becomes inconsistent with GitHub's actual state | Medium | Reconciliation with GitHub API (`GET /app/installations`) as periodic health check (configurable, e.g., every 10 minutes) |
