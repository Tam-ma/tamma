## Story

As a **GitHub App user who has installed Tamma**,
I want Tamma to execute autonomous development workflows within my repository's GitHub Actions runners,
so that issue processing runs in my own CI/CD environment with proper isolation, using my repository's Actions minutes.

## Overview

Implement a GitHub Actions worker mode where Tamma's engine runs inside a `workflow_dispatch`-triggered GitHub Actions workflow. The SaaS Coordinator (Story 1.5-12) dispatches work by triggering a workflow in the user's repository. The Actions workflow:

1. Receives inputs: `installation_id`, `issue_number`, `callback_url`
2. Obtains an installation token using `actions/create-github-app-token`
3. Runs `tamma processOneIssue` with the installation token
4. Reports the result (success/failure) back to the orchestrator via HTTP callback
5. Exits with appropriate exit codes (0 = success, 1 = failure, 78 = skip)

This story covers the workflow YAML template that users add to their repos, the worker entry point that runs inside Actions, and the result callback mechanism.

## Acceptance Criteria

1. GitHub Actions workflow template (`tamma.yml`) can be triggered via `workflow_dispatch` with inputs: `installation_id`, `issue_number`, `callback_url`
2. Worker acquires installation token using `actions/create-github-app-token` action within the workflow
3. Worker authenticates to Tamma SaaS API using `${{ secrets.TAMMA_API_KEY }}` (provisioned by Story 1.5-15) for LLM access and workflow status updates
4. Worker executes `tamma processOneIssue --issue <number>` using the installation token for Git operations and `TAMMA_API_KEY` for LLM inference via the Tamma API
5. Worker reports result to orchestrator via Tamma API (`POST /api/v1/workflows/{id}/result`) authenticated by `TAMMA_API_KEY`
6. Worker exits with correct exit codes: `0` (success), `1` (failure), `78` (skip/not-applicable)
7. Workflow template includes proper timeout (configurable, default 30 minutes), concurrency control (one job per issue), and error handling
8. Worker handles token expiry during long-running jobs by re-acquiring tokens
9. Unit tests cover: worker entry point, API key auth, callback reporting, exit code logic, token refresh
10. Integration tests cover: simulated workflow execution with mock GitHub Actions environment and mock Tamma API

### Requirements Context Summary

**Epic Context:** This story completes the SaaS execution pipeline: Story 1.5-11 handles auth, Story 1.5-12 dispatches work, and this story defines how work actually runs. By using GitHub Actions as the compute layer, Tamma's SaaS model doesn't require provisioning its own compute infrastructure — each installation uses its own repo's Actions minutes.

**Technical Context:** GitHub Actions `workflow_dispatch` allows external systems to trigger workflows via the API. The workflow receives inputs as `${{ github.event.inputs.* }}` values. The `actions/create-github-app-token` action (official GitHub action) converts App credentials into installation tokens within the Actions environment.

Exit code `78` follows the `sysexits.h` convention for `EX_CONFIG` — indicating the operation was not applicable (e.g., issue already has a PR, issue is closed, issue is not assigned to Tamma).

**Integration Points:**
- Story 1.5-12 (SaaS Coordinator): Dispatches `workflow_dispatch` events that trigger this workflow
- Story 1.5-11 (GitHub App Integration): Provides installation token mechanism
- Story 1-8 (Hybrid Architecture): Worker mode extends the existing worker architecture to GitHub Actions environment
- Story 1.5-14 (Multi-Tenant Task Routing): Workers report results back to the correct tenant's task queue

### Implementation Guidance

**Key Design Decisions:**

- **`workflow_dispatch` over self-hosted runners**: Using GitHub-hosted runners avoids infrastructure management. Each installation uses its own Actions minutes. Self-hosted runner support can be added later.
- **Callback over polling**: The worker POSTs results to the orchestrator rather than the orchestrator polling for completion. This is faster and avoids polling overhead for many concurrent installations.
- **Exit code 78 for skip**: Standard `sysexits.h` convention. GitHub Actions treats any non-zero exit as failure, but the callback payload includes the `skipped` status for the orchestrator to handle correctly.
- **Template, not auto-install**: Users add the workflow template to their repo manually (or via PR from Tamma). We don't automatically push workflow files — that would require `workflows` write permission which is a sensitive scope.

**Technical Specifications:**

- **Workflow file**: `.github/workflows/tamma.yml`
- **Required Actions permissions**: `contents: write`, `issues: write`, `pull-requests: write`
- **Default timeout**: 30 minutes
- **Concurrency group**: `tamma-issue-${{ inputs.issue_number }}` (prevents duplicate processing)
- **Callback payload**: JSON, max 1KB
- **Callback retry**: 3 attempts with exponential backoff (1s, 2s, 4s)

**Testing Strategy:**

- Unit tests mock the `TammaEngine` and test the CLI command wrapper
- Integration tests set `GITHUB_ACTIONS=true` and other Actions environment variables to simulate the Actions runtime
- No real Actions workflow runs in CI — test the entry point and callback independently
- Workflow YAML validated with `actionlint` in CI

### Risks and Mitigations

| Risk | Severity | Mitigation |
| ---- | -------- | ---------- |
| Users may not have GitHub Actions enabled or may have run limits | Medium | Check Actions availability during installation callback (Story 1.5-11). Provide clear error messaging. |
| Workflow template becomes outdated across user repos | Medium | Version the template. Tamma can open PRs to update the workflow file when a new version is released. |
| Callback URL may be unreachable from GitHub Actions runners | Medium | Callback URL must be publicly accessible. Document this requirement. Provide a fallback status check API the orchestrator can poll. |
| Long-running issues may exceed Actions timeout (6 hours max) | Low | Default 30-minute timeout is conservative. Large issues should be broken down. Timeout is configurable per-repo. |
